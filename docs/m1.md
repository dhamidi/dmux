# Milestone 1: Minimal Vertical Slice

## Goal

Running `dmux` with no arguments on a supported platform starts a terminal multiplexer session. If no server is running, dmux starts one, creates a new session, attaches to it, and the user lands in a single window running their default shell. A status line at the bottom of the screen shows the session name and window list.

Typing `exit` in the shell terminates that pane; because the session has no other panes, the session tears down and the client detaches. If that was the only session, the server exits.

## Acceptance criteria

From a fresh terminal:

1. `dmux` shows a shell prompt identical to running the shell directly. A status line appears at the bottom of the screen showing `[<session>] 0:<window>*`.
2. Commands typed into the shell execute and produce output identical to running the shell outside dmux (modulo `$TERM`, which dmux sets to `xterm-256color`).
3. Resizing the terminal window resizes the shell inside the pane; TUI apps reflow on SIGWINCH; the status line tracks the new width.
4. `exit` in the shell returns the user to their original terminal.
5. In a second terminal, `dmux kill-server` cleanly terminates the running dmux server.
6. On Windows, the first five items work with `dmux.exe` in recent Windows Terminal.
7. Heavy output in the pane (`yes`, `cat /var/log/...`) does not block the status line from updating or `kill-server` from being responsive — the per-pane goroutine isolates throughput-bound work.

## In scope

This is the full vertical slice: every package in the tree must have at least a minimal working implementation, even if several are stubs.

**Real implementations:** `cmd/dmux`, `proto`, `xio`, `socket`, `sockpath`, `platform`, `log`, `record`, `client`, `tty`, `server`, `session`, `pane`, `pty`, `vt`, `cmdq`, `cmd`, `cmd/newsession`, `cmd/attachsession`, `cmd/killserver`, `options` (small Table), `status`.

**Test infrastructure (behind `dmuxtest` build tag):** `dmuxtest`, `cmd/wait`, `cmd/assert`, `cmd/expect`, `cmd/at`, `cmd/testattach`, `cmd/testdetach`, `cmd/testsetrecorder`. M1 exit criteria require at least one passing scenario that covers new-session, pane-ready, at, wait output, assert screen.

**Minimal stubs:** `termin` (raw byte passthrough — see below), `keys` (types only, no bindings populated), `termcaps` (detection always returns `Unknown`), `termout` (full-repaint each frame, no diff).

## Out of scope

- Prefix key. There are no key bindings; everything the user types goes straight to the pane (via the M1 raw passthrough — see below).
- Multiple panes per window; splits.
- Multiple windows per session.
- Explicit detach via `Ctrl-B d`. The user detaches by exiting the shell or by `dmux kill-server`.
- Status line format strings (`#{...}`). M1's status content is hardcoded; M5 adds format expansion.
- `set-option` command. M1 reads options but does not let users change them.
- `.dmux.conf` loading.
- Copy mode, search, paste buffers.
- Scrollback *navigation*. Scrollback itself happens — libghostty-vt retains history by default and the vt interface exposes it — but M1 has no way to scroll into it. The user only ever sees the live screen.
- Capability detection. All clients are treated as `termcaps.Unknown`: basic SGR, legacy key encoding, no graphics, no KKP, no mouse.
- OSC 8 hyperlinks, sixel, kitty graphics, bracketed paste.
- Mouse support.

## M1 input path: raw passthrough

M1 does not parse input. Bytes from the user's terminal stdin reach the pane verbatim:

```
client tty.Read -> Input frame -> server -> pane.SendBytes(b []byte) -> pty.Write
```

`pane.SendBytes` is an M1-only shortcut on the pane goroutine's controlCh. It bypasses the encoder and writes raw bytes to the pty. M2-1 removes `SendBytes` and replaces it with the structured `SendKey` / `SendMouse` / `SendFocus` path that goes through the parser and encoder.

This works in M1 because the user's terminal and the pane app are both speaking xterm-style sequences and there are no bindings to interpose. It does not work for KKP, modifyOtherKeys, or anything where the encoding the pane app wants differs from what the user's terminal emits — but those features arrive in M2-1.

## Per-package work

**`cmd/dmux`** — flag parsing (`-S`, `-L`, `-V`, `-h`). Resolve socket path, dial via `socket.DialOrStart`. If the caller forked the server, that child runs `server.Run`; the parent becomes the client. Send Identify, then `Command{argv:["new-session"]}`, then `Command{argv:["attach-session"]}`. Run the client loop. Blank-import the three M1 command sub-packages so their `init` functions run.

**`proto`** — frame types for Identify, Input, Output, Resize, Command, CommandResult, Exit. `encoding/binary` for fixed-width fields; length-prefixed strings for variable data.

**`xio`** — buffered frame reader/writer over `io.Reader`/`io.Writer`. Correct, not optimized.

**`socket`** — AF_UNIX stream sockets on both Unix and Windows (requires Windows 10 build 17063+). `DialOrStart` implements the lock-and-retry dance from tmux's `client_get_lock`.

**`sockpath`** — tmux-style resolution: `-S` verbatim, else `$DMUX` prefix, else `<tmpdir>/dmux-<uid>/<label>` on Unix or `%LOCALAPPDATA%\dmux\<label>` on Windows. Both are filesystem paths consumed directly by AF_UNIX.

**`platform`** — daemonize: `fork` + `setsid` on Unix; `CreateProcess` with `DETACHED_PROCESS` on Windows. Signal handlers via `os/signal.Notify`. `HomeDir`, `CurrentUser`.

**`client`** — open tty, probe size, send `Identify` with `Unknown` profile, run three goroutines (stdin → Input, server frames → stdout/exit, resize → Resize), all derived from a per-client `context.Context`.

**`tty`** — raw mode on/off, read/write passthrough, size via `ioctl(TIOCGWINSZ)` or `GetConsoleScreenBufferInfo`, resize event via SIGWINCH or console control handler. Enables only the basics on attach (CSI for legacy mouse-off, etc.); KKP/SGR/focus enables come in M2-1.

**`termin`** — single-file stub that emits incoming bytes as opaque "raw input" events. No escape-sequence parsing. The server's M1 input handler unwraps and calls `pane.SendBytes`. M2-1 replaces this with the real parser and removes the SendBytes shortcut.

**`termout`** — naive renderer: each frame emits `\x1b[H\x1b[2J` (home + clear), then every cell of the pane grid with 24-bit SGR, then the status line. Inefficient but correct. M3 replaces with diff rendering.

**`termcaps`** — all five `Profile` variants defined; `Detect` always returns `Unknown`. `Features` struct exists but is read by no one yet.

**`server`** — main goroutine running `for { select { ... } }` over the events channel, signal channel, and ticker. Edge goroutines: per-client socket reader and writer, per-pane main goroutine + pane.pty.Read helper, signal handler, accept goroutine. Per-pane and per-client state is owned by their goroutines; the server's own state graph (session/window registry, client list, command queue, key tables) is owned by the main goroutine. See `internal/server/doc.go` and `docs/go-patterns.md`.

**`session`** — `Registry` with `NewSession`; one session has one window which has one pane. `Winlink` list exists but is always length 1.

**`pane`** — per-pane goroutine owns one `vt.Terminal`, one `pty.PTY`, one `vt.KeyEncoder`. M1 `controlCh` accepts only `sendBytes`, `resize`, `snapshot`, and `attachedClientsChanged`. The full `sendKey` / `sendMouse` / `capsProbe` set lands in M2-1. The pane goroutine pushes snapshots to the server on a timer (default 16ms) when the dirty flag is set. The pty.Read helper goroutine forwards bytes to the pane goroutine via a buffered channel; heavy output in this pane does not block any other pane or the main loop.

**`pty`** — `openpty` + fork + exec on Unix; `CreatePseudoConsole` + `CreateProcessW` on Windows using the modern API path (`ConptyReleasePseudoConsole` + read-until-EOF for exit detection, `ConptyClosePseudoConsoleTimeout(handle, 0)` for async close). Resize, Wait, Close.

**`vt`** — wazero runtime loads libghostty-vt wasm once at startup. Per-pane `Terminal` and `KeyEncoder` handles. The interface already includes `Viewport`, `LineRef`, and the history-addressing methods (`OldestLine`, `NewestLine`, `HistoryLen`, arbitrary-viewport `Snapshot`, bounded `Search`) — M1 doesn't call them, but they exist so M4 doesn't change the signature. `Graphics()` and `Hyperlinks()` return empty; M1 doesn't consume them.

**`keys`** — type definitions only. Tables exist but are not populated.

**`cmdq`** — `List` of pending `Item` values; `Drain` runs items via `cmd.Exec`. M1 commands all return `cmd.Ok` synchronously, so the continuation machinery (`cmd.Await`) is implemented but unexercised.

**`cmd`** — registry, `cmd.New[Args]` generic helper for typed commands, struct-tag-driven argument parser in the `args` sub-package, `ParseArgv` for argv-style input. The full text-syntax `Parse(text)` (semicolons, brace groups) can return "not supported" for M1.

**`cmd/newsession`, `cmd/attachsession`, `cmd/killserver`** — working implementations of the subsets documented in each sub-package's `doc.go`.

**`options`** — Table contains the four entries needed in M1: `default-shell` (session, string, default `/bin/sh` or `%COMSPEC%`), `default-terminal` (session, string, default `xterm-256color`), `status` (session, bool, default true), `status-position` (session, choice, default bottom). Server constructs the scope chain at startup; `newsession` reads `default-shell` via `sessionOptions.GetString("default-shell")`. No `set-option` command in M1.

**`status`** — renders one styled cell-row per attached client. M1 layout: `[session-name] index:window-name*`. Re-renders on session-state change events (window add/remove/rename, current-window switch, error message arrival, message timer expiry). Composed into the per-client frame by the server before termout renders it.

## Build order

Packages should come up in this order; each layer depends only on what's below it:

1. `proto`, `xio`, `sockpath`, `socket`, `platform`
2. `pty`, `vt`, `options`
3. `session`, `pane`, `status`
4. `cmd` + its three M1 sub-packages
5. `cmdq`
6. `tty`, `termin` (stub), `termout`, `termcaps` (stub), `keys` (stub)
7. `client`, `server`
8. `cmd/dmux`

No import cycles possible with this ordering.

## Known risks

**ConPTY edge cases on Windows.** Modern API (released Win10 22H2+) provides `ConptyReleasePseudoConsole` and `ConptyClosePseudoConsoleTimeout` which handle the historical exit-detection and shutdown-deadlock issues. The reference implementation is `github.com/UserExistsError/conpty` (we don't depend on it but use it as a reference). One remaining concern: ConPTY resize during the create-and-attach window can be ignored (microsoft/terminal#10400). `pty.Spawn` ensures both create and CreateProcess complete before accepting Resize calls.

**Windows daemonization.** No `fork` equivalent. `CreateProcess` with `DETACHED_PROCESS | CREATE_NEW_PROCESS_GROUP` and careful handle inheritance is required. Allocate extra time for `platform/daemonize_windows.go` — the most finicky Windows-specific task in M1.

**Per-pane snapshot rate-limit tuning.** The 16ms default is a guess. Heavy output may want it lower; very low-latency interactive feels may want 8ms. Make it a per-pane configurable internal field; M5's options system later exposes it as `pane-render-interval` or similar.

## Forward-compatibility carry-overs

Three interfaces in M1 are deliberately richer than M1 itself needs. If a reviewer asks "why is this here when nothing calls it?", removing it would force an interface change later.

1. **`vt.Viewport`, `vt.LineRef`, and the history methods on `vt.Terminal`.** M1 always reads the live screen. M4 (copy mode) reads arbitrary viewports. The two APIs have to be the same one; otherwise `pane.View()` has to change signature when copy mode lands. Keeping Viewport and LineRef in the M1 interface costs nothing at runtime and saves a refactor.

2. **`pane.Pane.viewport` as a field with no setter.** M4 adds `SetViewport`, `EnterCopyMode`, and `ExitCopyMode`, all of which mutate this field. The field exists in M1 so that snapshot reads it rather than always calling `LiveViewport()` directly.

3. **`termout.Frame` carrying a viewport fingerprint.** M1's full-repaint renderer doesn't diff and doesn't inspect the fingerprint. M3's diff renderer uses it to detect viewport changes (scrollback scrolls) and viewport-size changes (resize) and fall back to full repaint when the previous frame is not a usable diff base.

None of these implement anything; they widen the surface so later milestones slot in without disturbing callers.

## Risk no longer present

Earlier drafts flagged libghostty-vt-wasm as a single-point-of-failure dependency. It is not. The wasm32-freestanding target is officially supported by the Ghostty project (`zig build -Demit-lib-vt -Dtarget=wasm32-freestanding`), no emscripten required, and production consumers (`coder/ghostty-web`, the Dart bindings package) ship it. There is no need for an M0 "verify the wasm build works" milestone.

## Exit criteria

All seven acceptance items pass on Linux, macOS, and Windows. `go vet` and `go test ./...` are clean. No panics on expected paths. Status line updates within 100ms of session state change.
