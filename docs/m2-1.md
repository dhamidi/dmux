# Milestone 2-1: Input Parser and Capability Detection

The first half of M2. Pure infrastructure with minimal user-visible change. Replaces M1's raw-byte passthrough with the structured input pipeline that bindings (M2-2) and copy mode (M4) depend on.

## Goal

The dmux client detects which of the supported terminal profiles it is running in. The server parses input bytes from each client into structured key, mouse, focus, and paste events. Events going to the focused pane are re-encoded by libghostty-vt's key encoder so the pane app receives sequences appropriate to its current mode rather than verbatim copies of what the user's terminal emitted.

This is the milestone where M1's "treat the input stream as opaque bytes" stops being true.

## Acceptance criteria

1. Running `dmux` in Ghostty detects the profile as `Ghostty`. Same for xterm.js-modern, xterm.js-legacy, Windows Terminal. Verifiable via a debug command (`display-message #{client_termcaps_profile}` or equivalent — for M2-1 the value is logged at attach).
2. Typing keys in the shell still works exactly as in M1, with one substantive improvement: `Ctrl-Shift-<letter>` reaches an app running in the pane via the right encoding for the pane's mode, rather than being lost or doubled.
3. Bracketed paste from the user's terminal arrives at the pane as bracketed paste (not interleaved with regular keypresses).
4. The pane app's mode changes (enabling KKP, switching to alt-screen, requesting modifyOtherKeys) are tracked by libghostty-vt and reflected in subsequent encoder output without dmux explicitly observing them.
5. A late DA2 reply that arrives after the initial 100ms detection window updates the client's profile via `CapsUpdate` without disrupting the session.

No new commands, no bindings, no prefix key, no multi-window. Every typed character that wasn't typed in M1 still goes straight to the pane.

## In scope

**Real implementations replacing M1 stubs:**

- `termin` parser — full state machine for legacy + modifyOtherKeys + KKP key encodings, SGR mouse, bracketed paste, focus events, DA/DSR replies.
- `termcaps.Detect` — DA2 probe + KKP-detect probe + env-var checks; closed five-profile matrix.
- `vt.KeyEncoder` — was already in M1's interface but only used through the SendBytes shortcut; M2-1 makes it the only path.

**New methods / data flow:**

- `pane.SendKey(keys.Event)`, `pane.SendMouse(termin.MouseEvent)`, `pane.SendFocus(termin.FocusEvent)`. These replace `pane.SendBytes` as the input path.
- `pane.CapsProbe(probe termin.CapsResponse)` — for DA/DSR responses originating from the user's terminal that happen to traverse the pane's input stream (rare but real).
- Server-side per-client `termin.Parser` constructed at Identify time using the client's profile.
- `proto.CapsUpdate` frame handler in server: late capability information updates the client's parser and renderer.

**Removed:**

- `pane.SendBytes` (M1 shortcut). Bytes from the user's terminal no longer reach the pane verbatim; they go through parser + encoder.

## Out of scope

- Bindings, prefix key, detach, multi-window — all in M2-2.
- Status line — already shipped in M1 (only the message-overlay path is exercised in M2-1).
- Copy mode, scrollback navigation, splits, mouse-as-app-input — later milestones.

## Per-package work

**`termin`** — full parser. State machine recognising CSI, OSC, DCS, APC. Decodes:

- Legacy arrows / function keys (`ESC [ A`, `ESC O P`)
- Application-mode cursor keys
- modifyOtherKeys (`CSI 27;5;97~` and the formatOtherKeys variant `CSI 97;5u`)
- KKP (`CSI ... u` with progressive enhancement flags, alternate keys, release events)
- SGR mouse (`CSI < b ; x ; y M/m`); legacy X10 mouse not supported
- Bracketed paste wrappers
- Focus in/out (`CSI I` / `CSI O`)
- DA/DSR replies (routed to `termcaps` for late profile upgrades and to `pane.CapsProbe` for app-directed probes)

Parser is constructed for one of the `termcaps.Profile` values; the profile selects which key dialect is active. ESC-timeout disambiguation (default 25ms) is driven by `Tick` calls from the server's main loop.

**`termcaps`** — real `Detect`. Client runs at startup:

1. `TERM_PROGRAM=ghostty` → Ghostty.
2. `WT_SESSION` non-empty → WindowsTerminal.
3. Emit DA2 (`CSI > c`) and KKP-detect (`CSI ? u`). Wait up to 100ms.
   - Both replied → XTermJSModern.
   - Only DA2 replied → XTermJSLegacy.
4. Otherwise → Unknown.

Result is included in the Identify frame. Later DA2/KKP replies that change the conclusion produce a `CapsUpdate` frame to the server.

**`pane`** — `SendBytes` is removed. Three new control commands on the pane goroutine's `controlCh`:

- `sendKey{event keys.Event}` — encoder.Encode(event) → pty.Write
- `sendMouse{event termin.MouseEvent}` — encoded via libghostty-vt's mouse encoder, configured from the pane's terminal modes
- `sendFocus{event termin.FocusEvent}` — encoded via libghostty-vt's focus encoder
- `capsProbe{probe ProbeKind}` — pane goroutine looks up its `attachedProfiles` snapshot, composes the appropriate reply (or "no" for unsupported features), writes to pty input

**`server`** — per-client construction at `Identify` handler now creates a real `termin.Parser` from the client's profile. The Input-frame handler feeds bytes into the parser, gets events, and routes them straight to the focused pane (no key-table check yet — bindings land in M2-2). Mouse and focus events likewise go to the focused pane.

`CapsUpdate` frame handler: replace the parser and renderer for the affected client.

**`cmd`** — no new commands. The framework is unchanged.

**`vt`** — no interface change. M2-1 is the first milestone that actually exercises `KeyEncoder()` and the mouse/focus encoders.

## Why this is its own milestone

Three reasons:

1. **The parser is large.** Roughly the size of tmux's `tty-keys.c` (~2,800 lines in C; less in Go but still substantial). Bundling it with bindings and detach makes M2 a step-change in scope where it should be two predictable steps.

2. **Behavior change is catchable.** Going from "raw passthrough" to "parsed-and-re-encoded" should produce identical user experience for normal keys, plus better handling of `Ctrl-Shift-letter` and friends. If anything regresses, M2-1 is where it surfaces, before the binding logic is added.

3. **The continuation infrastructure (`cmd.Await`) doesn't ship until M2-2** — M2-1 has no async commands. Splitting M2 means the cmdq changes land closer to the command (`command-prompt`) that motivates them.

## Known risks

**Escape vs. Alt disambiguation.** A lone Escape looks like the start of an escape sequence. The 25ms timeout works on local terminals; on slow/lossy connections, an Escape press can be misclassified as the start of a sequence. Allocate time for a borderline-timing test suite.

**KKP progressive enhancement.** The protocol has five flag bits that combine, and the parser must produce events that match what the encoder will emit when fed the same logical key. Use Ghostty's KKP test vectors as ground truth.

**Late capability updates and parser state.** When `CapsUpdate` arrives (e.g. an xterm.js-modern client whose KKP probe was slow), the parser is reconstructed mid-stream. Any in-flight escape sequence is dropped on the floor. This is acceptable but should be logged so it's diagnosable.

## Exit criteria

All five acceptance items pass on Linux, macOS, and Windows against the three target terminals. No regression vs. M1: typing in the shell still works, output still appears, resize still reflows.
