# Milestone 2-2: Prefix Key, Detach/Attach, Multiple Windows

The second half of M2. With M2-1's parser and capability detection in place, this is where dmux starts to behave like tmux from the user's perspective.

## Goal

The prefix key works. Users can press `Ctrl-B d` to detach, `Ctrl-B c` to create a new window, `Ctrl-B n` to cycle, and `Ctrl-B ?` to list bindings. The default binding set loads at server startup by parsing `bind-key` command strings — the same syntax users will write in `.dmux.conf` later. Sessions survive client detach; reattaching restores the running state.

## Acceptance criteria

1. `Ctrl-B d` detaches the client; the session keeps running on the server.
2. `dmux attach` (or `dmux attach-session`) from any terminal reattaches; long-running processes inside are preserved.
3. `Ctrl-B c` creates a new window running the user's shell.
4. `Ctrl-B n` and `Ctrl-B p` cycle forward and backward through windows.
5. `Ctrl-B 0` through `Ctrl-B 9` select a window by index.
6. `Ctrl-B ?` lists all bindings; pressing any key dismisses the list.
7. `Ctrl-B :` opens a command prompt at the bottom of the screen; typing a command and pressing Enter executes it.
8. `Ctrl-B Ctrl-B` sends a literal `Ctrl-B` to the focused pane (the `send-prefix` binding).
9. Unbound keys pass through to the focused pane (M2-1 behaviour preserved).
10. The status line (from M1) updates to reflect the current window list and active window. When the active window changes, the status line updates within one render tick.

## In scope

**New commands** (one sub-package each):

- `detach-client`
- `send-prefix`
- `bind-key`, `unbind-key`
- `list-keys`
- `new-window`, `kill-window`
- `next-window`, `previous-window`, `select-window`
- `command-prompt` — the interactive `:` command; first user of `cmd.Await`
- `display-message`

**New infrastructure:**

- Populated `keys` tables: `root` (active when prefix is inactive), `prefix` (active for one keypress after the prefix matches), `command-prompt` (active during a `command-prompt` modal).
- Prefix state machine on each server-side Client struct: which key table is active, repeat-key debouncing, table reset after a non-repeating binding fires.
- `cmd.Await` continuations: the queue can park a command and resume on a channel. Continuations replace tmux's `CMDQ_WAITING` flag mechanism. See `internal/cmdq/doc.go`.
- Multi-window support in `session`: real Winlink list with indexed insertion, current-window tracking, lowest-unused-index allocation.
- `pane.AttachedClients(profiles []termcaps.Profile)` is exercised for real now: multiple clients can attach to one session and each has its own current-window state, but the panes' capability answers are the most-restrictive across attached clients.
- Default binding set loaded at server startup by parsing a static slice of bind-key strings (mirrors tmux's `key_bindings_init`).

## Out of scope

Splits and layouts (M3). Mouse-driven binding dispatch (M3, alongside layouts). Copy mode (M4). Hooks, format strings, options-as-commands, `.dmux.conf` (M5). Status-line format strings stay disabled — the M1 hardcoded layout is enough.

## Per-package work

**`keys`** — real `KeyCode` normalization (Shift on printable letters folded into character case). Tables are now populated; lookup is a single map access. Repeat-binding flag (`-r` in tmux) keeps the prefix table active for a short window after a matching binding fires, so `Ctrl-B C-Up C-Up C-Up` works as three resize-pane invocations.

**`cmd/bind-key`, `cmd/unbind-key`** — `bind-key` parses the inner command text as a `*cmd.List` and installs it in the named table. `unbind-key` removes.

**`cmd/detach-client`** — cancels the calling client's `context.Context`. The cancellation cascade (writer goroutine flushes pending Output, sends `Exit{Detached}`, closes socket; reader exits on socket close) handles the rest. The session continues running.

**`cmd/command-prompt`** — first command using `cmd.Await`:

```go
func exec(item *cmd.Item, _ *Args) cmd.Result {
    return cmd.Await(item.Prompt(":"), func(line string) cmd.Result {
        if line == "" {
            return cmd.Ok
        }
        list, err := cmd.Parse(line)
        if err != nil {
            return cmd.Err(err)
        }
        return cmd.RunList(list)
    })
}
```

The `item.Prompt` helper installs a one-line overlay at the bottom (above the status line if present, replacing it otherwise — TBD in M2-2 design) and switches the client's active key table to `command-prompt`. Enter fires the channel with the typed line; Esc cancels by cancelling the item's context, which causes the await goroutine to drop without running the continuation.

**`cmd/new-window`, `cmd/kill-window`** — manipulate the session's Winlink list. Window-index allocation: lowest unused non-negative integer. The `base-index` option from tmux is deferred to M5.

**`cmd/next-window`, `previous-window`, `select-window`** — change the calling client's `currentWindow`; the next render produces an `Output` reflecting the new window's focused pane. Status line updates because session state changed.

**`server`** — the real key-dispatch path:

1. M2-1 parser emits a `keys.Event`.
2. Build a `keys.KeyCode` (press and repeat actions only; release ignored until copy mode).
3. Look up `client.keyTable[keyCode]`.
4. Hit: append the binding's cmdlist to the client's cmdq; reset `keyTable` to `root` (or keep at `prefix` briefly if the binding had `-r`).
5. Miss: forward the event to the focused pane via `Pane.SendKey`.

Default bindings are parsed at startup before the listener accepts clients. If `bind-key` is broken, the server fails to start with a clear error — useful regression detector.

**`cmdq`** — wires `cmd.Await` into the events loop: when `Drain` sees an `Await`-typed Result, spawns a one-shot fan-in goroutine that selects on the awaited channel and on the item's context. Channel value → push `ContinuationReady` event; context cancel → drop awaiter without running continuation. Main loop receives `ContinuationReady`, runs the continuation; result is itself either Done, Err, or another Await.

**`session`** — Winlink list operations: insert, remove, find-by-index, iterate-in-order. Per-session `currentWinlink` for each attached client (or shared per-session — M2-2 picks shared, matching tmux).

**`pane`** — capability mediation now operates on a real `attachedProfiles` snapshot that can have more than one entry. When one Ghostty client and one Windows Terminal client are attached to the same session, panes answer probes conservatively. The "what to do when client list changes" path is exercised: server pushes an `attachedClientsChanged` control message to every pane in the affected session.

**`status`** — the M1 status line gains real content. Window list now reflects `session.Winlinks` ordered by index, with the active window marked. Error-message overlay (already in M1) is now invoked from `display-message`.

## Key design decisions

**One prefix implementation, not two.** The prefix key is a regular binding in the `root` table that switches the client's active key table to `prefix` and arranges for it to revert to `root` on the next non-repeating keypress. Everything else in the `prefix` table is a regular binding. There is no dedicated "prefix mode" code path. Same as tmux.

**bind-key is self-hosting.** The default bindings are installed by parsing bind-key command strings at startup. This exercises the full parser end-to-end before any user input arrives; if anything is broken, startup fails loudly. It also means defaults are testable by running commands.

**command-prompt is the template for async commands.** The pattern — `Exec` returns `cmd.Await(item.Prompt(...), continuation)` — is what `confirm-before` (M3), `menu` (post-M5), and `choose-tree` (M4) all follow. Getting it right here pays forward.

**Multi-client capability mediation is sticky.** Once a session has clients attached, panes answer capability probes based on the most-restrictive profile present. When clients leave, answers don't widen retroactively (no terminal protocol allows that). Document and live with it; users who want the "best" answer should detach all conservative clients before running capability-sensitive apps.

## Known risks

**Default-binding parsing failures at startup.** Because `bind-key` is self-hosting, a typo in a default binding string crashes the server before any client connects. Good for catching bugs, terrible for shipping a buggy patch. Make startup-time binding errors test-covered before shipping M2-2.

**Prefix-table reset under repeat bindings.** `Ctrl-B C-Up C-Up` should resize twice; `Ctrl-B C-Up x` should resize once and then send `x` to the pane (because `-r` only extends the prefix table for repeated matches of the same binding, not for unrelated keypresses). The state machine here has subtle cases. Borrow tmux's tests if possible.

**Multi-client session resize.** M2-2 ships actual multi-client support. When two clients of different sizes attach to a session, the session sizes to the smallest (the `window-size` option, defaulting to `smallest`, controls this; the option is stub-set in M2-2 even though `set-option` doesn't ship until M5). The pane resize messages cascade through every pane in every window — keep an eye on the cost.

**Continuation lifetime under client detach.** When a `command-prompt` is open and the client detaches via socket-close, the await fan-in goroutine sees the context cancel and drops the continuation. The cmdq's parked item is unreachable; GC collects. Add an integration test for "open command-prompt, kill the client, verify no goroutine leak."

## Exit criteria

All ten acceptance items pass on Linux, macOS, and Windows against the three target terminals: Ghostty, recent xterm.js (VS Code 1.109+), and recent Windows Terminal. No regression vs. M2-1: parser still correct, capability detection still right.
