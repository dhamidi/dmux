# Milestone 3: Splits and Layouts

Details will firm up as M2 ships. The scope below is the expected shape.

## Goal

A window contains multiple panes arranged in a layout tree. Users can split panes horizontally or vertically, navigate between them, resize them, and close them individually. The M1 full-repaint renderer is replaced with a real diff renderer; multiple panes without diff rendering is unusable.

## Acceptance criteria

1. `Ctrl-B %` splits the active pane horizontally; `Ctrl-B "` splits vertically.
2. `Ctrl-B o` cycles to the next pane; `Ctrl-B ;` returns to the previously active pane.
3. `Ctrl-B <arrow>` moves pane focus directionally.
4. `Ctrl-B x` kills the active pane after confirmation.
5. `Ctrl-B z` toggles pane zoom (maximise active pane to fill the window).
6. `Ctrl-B Ctrl-<arrow>` resizes the active pane by one cell; `Ctrl-B Alt-<arrow>` resizes by five.
7. `Ctrl-B Space` cycles through the five named layouts.
8. Mouse click on a pane selects it; click on a pane border starts a drag-resize.
9. Redraw performance: a full-screen pane running `vim` or `top` renders without visible tearing on all three target terminals.

## In scope

**New commands:**

- `split-window` (with `-h` / `-v`, `-p` percentage, `-l` size)
- `select-pane` (with `-L/-R/-U/-D` directional, `-t` explicit, `-m/-M` mark)
- `kill-pane`, `swap-pane`, `rotate-window`
- `resize-pane` (directional and `-Z` zoom)
- `break-pane`, `join-pane`
- `select-layout`, `next-layout`, `previous-layout`
- `display-panes` (the pane-number overlay)
- `confirm-before` (second async-command template; needed for `kill-pane`)

**New infrastructure:**

- Layout tree in `session`: mirror of tmux's `layout_cell` with `LEFTRIGHT`, `TOPBOTTOM`, and `WINDOWPANE` cell types.
- Pane border rendering in `termout`; requires geometry from the layout tree.
- Real diff rendering in `termout`: per-client previous-frame cache, emit only the minimum sequence to transform the old frame into the new one. Based on tmux's `tty-draw.c` diff loop.
- Mouse event routing: `termin` already parses SGR mouse in M2; now `server` dispatches mouse events to the active key table (mouse-click keys exist in tmux's default bindings), and unbound events go to the pane's app if mouse reporting is on for that pane.
- Pane resize propagation: layout change → `pty.Resize` each pane → `vt.Resize` each terminal → next render reflects new geometry.

## Out of scope

Copy mode, paste buffers (M4). Status line, options, config file (M5). Hooks, format strings, grouped sessions (post-M5).

## Open questions

**Layout tree allocation.** Every resize recomputes every cell's dimensions. tmux does this cheaply because it's C with manual memory management; the Go equivalent needs profiling on deep layouts (eight or more panes). If allocation dominates, cell geometry becomes a struct stored by value in a slice keyed on layout-cell ID, rather than a pointer tree.

**Pane border degradation.** When a pane shrinks to one cell high, borders eat the whole pane. tmux's behavior is to hide the pane's content and show only a status indicator. We match this, but the status indicator itself needs design (just a dim background? a centered label?). Defer the design decision until we've seen it in practice during M3 development.

**Mouse-event consumption rules.** When a pane's app has mouse reporting on and a key-binding also matches a mouse event, which wins? tmux's rule: bindings win unless the binding explicitly forwards with `send-keys -M`. Implementing this requires the `send-keys -M` flag, which means `send-keys` gets added in M3, not deferred.

## Risks

**Diff-rendering correctness.** The per-client previous-frame cache has to stay in sync with what the client actually rendered. A dropped or corrupted Output frame would desync the cache; on the next render the diff produces garbage. Mitigation: Output frames are sequenced; the client acks periodically; the server falls back to a full repaint on ack timeout. This is added complexity in `proto` and `server`.

**Zoom state.** `Ctrl-B z` is a per-window flag, not a layout change. The active pane renders full-window while zoomed, but the underlying layout is preserved and restored on toggle. Getting this right (and not corrupting the layout on resize-while-zoomed) requires care in `session`.

**ConPTY resize on Windows.** Resizing many panes rapidly on Windows sometimes loses SIGWINCH-equivalent events. Budget a `pty_windows.go` workaround that polls size on write and re-sends on mismatch.
