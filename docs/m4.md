# Milestone 4: Copy Mode, Scrollback, and Paste Buffers

Speculative shape; will firm up during M3.

## Goal

Copy mode lets the user navigate a pane's scrollback, select text, and yank it into a paste buffer. Paste buffers are first-class server state; `paste-buffer` writes back into a pane. On capable clients, OSC 52 bridges paste buffers to the system clipboard.

## Acceptance criteria

1. `Ctrl-B [` enters copy mode in the active pane; the pane detaches from its pty input and becomes navigable.
2. Emacs-style movement bindings work: `C-b`/`C-f` by char, `M-b`/`M-f` by word, `C-p`/`C-n` by line, `g`/`G` top/bottom, `/`/`?` search forward/backward.
3. A vi-style binding set is available when `mode-keys` is `vi` (M5 config).
4. `Space` starts a selection; `Enter` yanks into the `default` paste buffer and exits copy mode.
5. `Ctrl-B ]` pastes the default buffer into the active pane.
6. `Ctrl-B =` lists paste buffers and lets the user pick one to paste.
7. On Ghostty and xterm.js-modern, yanks also write to the system clipboard via OSC 52 when `set-clipboard` is on.
8. Scrollback depth is at least 2000 lines per pane (configurable; default matches tmux).

## In scope

**New commands:**

- `copy-mode` (enter the mode)
- `send-keys` with `-X` (dispatch a copy-mode action by name)
- `set-buffer`, `show-buffer`, `list-buffers`, `delete-buffer`
- `paste-buffer` (with `-p` to use bracketed paste if the pane supports it)
- `load-buffer`, `save-buffer` (file I/O)
- `choose-buffer` (interactive picker; third async-command template)

**New infrastructure:**

- Copy-mode machinery in `pane`: a pane in copy mode diverts input-stream events to a local editor state rather than encoding for the pty. Rendering stays the same — termout reads a `vt.Grid` — but the pane's current `vt.Viewport` moves into scrollback instead of staying on the live screen. An overlay shows the copy-mode cursor and selection; that overlay is drawn by termout on top of the grid.
- Viewport-manipulation methods on `pane.Pane`: `SetViewport`, `EnterCopyMode`, `ExitCopyMode`, `ResetViewportToLive`. These were reserved in M1; M4 implements them.
- Scrollback navigation: `vt.Terminal` already exposes `OldestLine`, `NewestLine`, `Snapshot(Viewport)`, and a bounded `Search` iterator from M1. M4 is the first caller that uses any of them. No `vt` interface changes.
- Paste buffer store in `server`: a named-buffer map with tmux-style LRU semantics for the unnamed slot.
- Mode-trees in `keys`: named tables `copy-mode` and `copy-mode-vi`, installed from defaults at startup just like the prefix table.

## Out of scope

Rectangle selection, regex search (both are tmux features but behind the main M4 goal; defer to M4.5 if needed). Mouse drag-selection (depends on M3 mouse support and can be done in M4 or later). External paste buffer sync via files (`buffer-limit` beyond the default).

## Open questions

**libghostty-vt scrollback reader semantics.** The `vt.Terminal.Snapshot(Viewport)` interface was fixed in M1, but its implementation in M4 needs to decide: does a Snapshot of scrollback lines see live content (the pane's app could be writing to the scrollback region via alt-screen exit, for instance) or a stable view? Prefer live reads — they're cheaper and match libghostty-vt's natural behavior — and require copy mode to refetch before rendering each frame.

**Selection coordinate system.** Selections that span scrollback and live screen need one coordinate system across both. Use `(LineRef, col)` with the `vt.LineRef` introduced in M1. Selections hold two refs; if either becomes invalid (line dropped from scrollback, or invalidated by Resize), the selection is cleared.

**OSC 52 size limits.** Some terminals cap OSC 52 payloads. Ghostty allows large; xterm.js tends to cap around 100KB. Document the cap per profile in `termcaps` and truncate (with a warning) when yanking larger selections.

## Risks

**Copy-mode state lives in `pane`, not `server`.** This is a departure from tmux, which has it split across `window-copy.c` and `mode-tree.c`. Keeping it in `pane` makes the "this pane is special right now" state easier to reason about and fits the narrow interface pattern — but it means `pane` grows noticeably in M4. Budget for the refactor and keep copy-mode state in its own file within the package.

**Search performance on deep scrollback.** Plain substring search over 10,000 lines should still be fast in Go; regex search (if added later) may need tighter per-tick bounds. The `vt.Search` iterator signature reserved in M1 already takes a `maxLines` budget per call, so a slow query cannot stall the server main loop — M4's job is to pick a sensible per-tick budget and resume across ticks.
