# Milestone 5: Status Line, Options, and Configuration

Speculative shape; firms up during M4.

## Goal

A configurable status line at the bottom (or top) of each attached client's screen shows session name, window list, and user-defined fields. The options system lets users configure behaviour per server, per session, and per window. A `.dmux.conf` file runs at server startup. Format strings (`#{...}`) work everywhere they work in tmux.

This milestone is where dmux stops looking like a demo and starts looking like tmux.

## Acceptance criteria

1. A status line appears at the bottom of the attached client's screen showing `[<session>] <window-index>:<window-name>* <window-index>:<window-name> ...` with the active window marked with `*`.
2. `set-option -g status-position top` moves the status line to the top.
3. `set-option -g status-right '#[fg=green]%H:%M'` renders the current time in green on the right.
4. `show-options -g` lists all server options; `show-options -w` lists window options; `show-options` alone lists session options.
5. A `~/.dmux.conf` file with one command per line is loaded when the server starts.
6. `source-file path` loads an additional config file at runtime.
7. `set-hook -g session-created 'display "session started!"'` fires the message when a new session is created.
8. Format strings in `display-message`, status line, and rename commands expand correctly: `#{session_name}`, `#{window_index}`, `#{pane_current_command}`, `#{?condition,yes,no}`, etc.

## In scope

**New commands:**

- `set-option`, `set-window-option`, `show-options`, `show-window-options`
- `set-environment`, `show-environment`
- `set-hook`, `show-hooks`
- `source-file`
- `rename-session`, `rename-window` (use format expansion)
- `refresh-client` (redraw on config change)
- `if-shell` with format-string condition support

**New infrastructure:**

- `options` package: a new top-level package owning the option table (mirrors tmux's `options.c`/`options-table.c`), with per-server, per-session, per-window, per-pane scopes. Option lookup walks the scope chain.
- `format` package: expander for `#{...}` with the subset of tmux's format variables that the above commands need. Formats are evaluated on demand inside commands and the status-line renderer.
- `status` package: renders the status line. Runs on the server, output composed into the termout frame for each client. The status line is a thin wrapper pane with its own grid.
- `hooks` package: hook registry plus firing logic. Hooks are cmd.Lists installed under named events and run through the normal command queue.
- Config loader in `server`: reads `~/.dmux.conf` at startup, feeds each line through `cmd.Parse`, enqueues on the global cmdq.

## Out of scope

tmux's control-mode (post-M5 — it's a large subsystem in its own right). Grouped sessions. Menus and popups (visually similar to status line but structurally different; a separate milestone if we want them). tmux's `choose-tree` (M4 added enough interactive pickers for now).

## Open questions

**Scope of format strings.** tmux's format grammar is large (conditionals, string manipulation, arithmetic, time formatting). The minimum viable subset is: simple variable substitution (`#{session_name}`), the conditional (`#{?a,b,c}`), and basic string literals with `#[...]` style directives. Defer arithmetic, loops, and string-manipulation operators to a later milestone; document the subset clearly in the `format` package doc.

**Options table as source of truth vs. code.** tmux generates its option table from a single `options-table.c` file. We match that pattern: one Go file lists every option with its type, default, and scope. Adding an option = one table entry. Resist the urge to scatter option lookups through the codebase — consult the table via `options.Get(scope, name)` and respect scope inheritance.

**Hook timing.** Hooks fire synchronously through cmdq. A hook that takes time blocks the queue it runs on. This matches tmux's behavior and is the right default, but we should document it so users don't put slow shell commands in status-left.

## Risks

**Status-line redraw cost.** The status line redraws on a timer (default every 15 seconds) and on every event that could affect it (pane output for activity monitoring, window rename, session switch). Naively this causes full screen redraws — use the diff renderer from M3 to keep it cheap.

**Format-string injection.** User-supplied format strings in command arguments could theoretically construct commands that evade `cmd.Parse`'s validation. Format expansion happens *after* parsing, so a format result is not reparsed as commands — but document this explicitly so nobody adds a "re-parse after expansion" mode without thinking through the consequences.

**Config errors on startup.** A broken `~/.dmux.conf` shouldn't prevent the server from starting. Collect errors, display them in the first client's status area (or via `display-message` if status isn't visible yet), and continue.
